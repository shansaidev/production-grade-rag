How We Built a Production RAG System at Acme Engineering

Background and Motivation

Acme's engineering organisation grew from 20 to 85 engineers in 18 months. That growth is exciting, but it created a knowledge problem: new engineers spent their first two weeks reading through Confluence pages, Notion docs, Slack threads, and Google Docs trying to understand how things worked. Senior engineers were spending 3–4 hours per week answering questions that were already documented somewhere.

We decided to build an internal knowledge assistant that could answer engineering questions from our existing documents. This article describes what we built, what we got wrong, and what we would do differently.

Architecture Overview

After evaluating several approaches, we settled on a hybrid Retrieval-Augmented Generation (RAG) architecture. The high-level components are:

  • Document ingestion pipeline — parses PDF, DOCX, HTML, and Confluence exports into structured chunks
  • Dual embedding — dense embeddings (768-dim, nomic-embed-text) and sparse SPLADE embeddings stored in Qdrant
  • Hybrid retrieval — dense + sparse search fused with Reciprocal Rank Fusion (RRF), giving 14% recall improvement over dense-only
  • Three-stage reranker — FlashRank cross-encoder → rule-based recency/authority boost → MMR diversity filter
  • LangGraph reasoning engine — stateful graph with 3-agent system (Retriever, Reasoner, Verifier)
  • Semantic cache — Redis Stack with redisvl, reducing LLM calls by 34% within the first month

What We Measured

We used RAGAS to evaluate our system at each major milestone. The table below shows our scores across four evaluation runs:

Milestone Faithfulness Answer Relevancy Context Precision Cache Hit Rate
v0.1 — Naive RAG (no reranker) 0.61 0.68 0.54 N/A
v0.2 — Added FlashRank reranker 0.72 0.71 0.73 N/A
v0.3 — Added semantic cache 0.79 0.75 0.78 22%
v0.4 — Production (MMR + rule boost) 0.84 0.78 0.82 34%

The biggest single improvement was adding the FlashRank cross-encoder reranker with a 10:1 N:K ratio (50 candidates → top 5). Context precision jumped from 0.54 to 0.73 — the reranker filtered out a lot of plausible-sounding but actually irrelevant chunks that were hurting faithfulness.

The Biggest Mistake We Made

We spent two weeks building a beautiful LangGraph agent orchestration system before we had confirmed that basic retrieval quality was good enough. When we finally ran RAGAS, our baseline faithfulness was 0.61 — well below our target of 0.80. We had to pause agent work and spend another week fixing chunking.

The lesson: retrieval quality is the foundation. An agent system on top of bad retrieval is just a faster way to produce confident wrong answers. Measure RAGAS on a basic pipeline first. Only add agents when your context precision and faithfulness are already acceptable.

Chunking: The Underrated Foundation

Our first attempt at chunking used LangChain's RecursiveCharacterTextSplitter with a 500-token limit. It produced chunks like this:

"...should be submitted via the HR Portal at least 30 days in advance. For unforeseen events, employees should | Column A | Column B | 1 | 2 | 3 | 4 | notify HR as soon as practicable..."

We had split a table mid-row and merged it with surrounding prose. The resulting chunk was unchunkable by the embedding model — neither the table data nor the surrounding text was retrievable with any query.

Our custom SmartChunker addresses this with one non-negotiable rule: tables are always one chunk, never split. We detect tables by their bounding box in pdfplumber and exclude them from prose splitting. This change alone improved our context recall by 0.12.

What Is Next

We are planning three improvements for Q3 2026:

  1. Corrective RAG (CRAG) — when retrieval confidence is below 0.6, fall back to a web search to supplement context
  2. Parent-child chunking — index small chunks (128 tokens) for retrieval precision, but serve the parent paragraph (512 tokens) to the LLM
  3. Streaming responses — the current synchronous API returns the full answer at once; streaming will reduce perceived latency from 2.1s to first-token latency of ~200ms

If you are building something similar, the most important advice we can give is: measure first, build second. RAGAS is free, takes 10 minutes to set up, and will tell you exactly where your system is failing before you spend a week building agents on top of broken retrieval.
